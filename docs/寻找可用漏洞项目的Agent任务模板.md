# 寻找可用漏洞项目的 Agent 任务模板

## 1. 文档目的

这份文档用于直接交给别的 agent，帮助其为当前工具筛选 **可用于跨语言供应链漏洞可达/可触发检测** 的真实项目与漏洞。

这里的“可用”不是泛指“有漏洞”，而是指：

- 能定位到 **真实 Rust 项目**
- 项目依赖了 **存在已知 CVE 的 C / C++ / native 组件**
- 当前工具有希望基于 **CPG + 规则** 检测出“可达”或“可触发”
- 最终结果要 **可解释**，而不是只给一个模糊猜测

本模板偏向 **n-day 漏洞检测**，不是 0day 挖掘任务。

---

## 2. 你要帮我找什么

请帮我寻找一批 **Rust 真实项目 + 已知 CVE + native 组件依赖链** 的组合，要求这些组合适合接入当前静态分析工具。

目标不是“找到最多漏洞”，而是找到 **最适合工具落地、最容易形成高质量规则、最容易解释为什么可触发** 的案例。

优先寻找以下类型：

- Rust 项目通过 `*-sys` crate、FFI crate、bundled/vendored 方式依赖 C 组件
- 漏洞存在于底层 native 组件，而 Rust 侧能看到明显的调用入口或配置逻辑
- 漏洞触发条件可以用“版本 + 调用路径 + 参数/分支/状态/特征约束”来描述
- 最终能形成“为什么这个项目可触发”的清晰证据链

---

## 3. 当前工具的能力边界

请按下面的现实边界筛项目，不要按理想化条件筛选。

### 3.1 当前工具核心依赖

当前工具的核心是：

- Rust 侧 `Cargo` 依赖与版本解析
- Rust CPG 生成与导入
- 基于规则的 sink / 条件匹配
- 结合调用路径、版本、条件证据，给出可达/可触发结论

因此，**没有可生成的 Rust CPG，就不算合格候选**。

### 3.2 当前工具更擅长的条件类型

更适合当前工具的条件包括：

- 脆弱版本存在
- Rust 侧能走到某个 sink / wrapper / builder / decode / compile / open / parse 路径
- Rust 代码中能看到：
  - builder flag 链
  - 布尔开关
  - `Option` / `match` / `if` 分支
  - 文件路径、字节流、字符串、pattern、header buffer 等输入对象传递
  - 简单长度 / 容量 / size 约束
  - API 序列与对象状态前置条件
  - feature / build mode / runtime mode

### 3.3 当前工具不适合的案例

请尽量避开以下情况：

- 只有 native 二进制，没有 Rust 源码上下文
- Rust 项目无法生成 CPG，或编译上下文极难恢复
- 漏洞触发条件全部藏在底层复杂文件格式内部，Rust 侧完全看不到任何前置配置
- 只能靠动态执行或环境竞态才能判断，静态上几乎没有证据
- 无法确定实际使用的是哪个 native 版本
- 只知道“调用过某库”，但根本无法定位到具体高价值 sink

---

## 4. 候选项目的硬性要求

下面这些要求是 **必须满足** 的，缺一项就不要推荐为主候选。

### 4.1 项目要求

- 必须是 **真实 Rust 项目**，不是只包含最小 PoC 的实验仓库
- 源码可获得，最好能拿到具体 tag / release / commit
- 有 `Cargo.toml`，最好有 `Cargo.lock`
- 至少能运行 `cargo metadata`
- 最好能恢复出 CPG 需要的 target / feature / crate 入口

### 4.2 漏洞要求

- 必须是 **公开已知 CVE**
- 漏洞位于 C / C++ / native 组件，不是纯 Rust 逻辑漏洞
- 漏洞版本范围尽量明确
- 最好能查到修复版本、受影响版本、触发条件说明
- 最好有官方公告、issue、commit、PoC、分析文章中的任意一种辅助材料

### 4.3 依赖链要求

- 必须能明确说明：`Rust 项目 -> Rust crate -> sys/ffi crate -> native 组件`
- 最好能判断：
  - bundled / vendored
  - system library
  - feature 控制是否启用该组件
  - 实际锁定版本

### 4.4 检测可落地要求

- 必须能指出至少一个 **Rust 侧高价值 sink / wrapper / entry**
- 必须能描述出这条漏洞的 **触发条件**
- 这些条件里至少有一部分要能在 Rust 代码中看到
- 最终要能解释“为什么这个项目可能触发”，而不是只说“它依赖了脆弱库”

---

## 5. 优先级更高的候选特征

如果多个候选都满足硬性要求，优先推荐下面这些：

### 5.1 版本证据清晰

- `Cargo.lock` 里能直接看到脆弱 crate / sys crate 版本
- 或者 vendored source 目录里能直接看出 native 版本
- 或者构建脚本 / docs 明确写了 bundled 组件版本

### 5.2 Rust 侧可见性高

- 能在 Rust 代码中直接看到危险分支或配置
- 能看到 sink 前的 builder / flag / option / mode 配置
- 能看到输入对象如何进入 sink
- 能看到“必须先调 A，再调 B”的 API 序列

### 5.3 适合静态规则表达

优先选这类漏洞条件：

- `if is_webp { decoder.decode() }`
- `File::open(path)` -> `read_to_end` -> `Decoder::new(&bytes)` -> `.decode()`
- `inflateGetHeader(&mut header)` + `inflate()` + `extra_max < assumed_extra_len`
- `RegexBuilder` / `PCRE2 builder` 上多个 flag 同时成立
- Git / archive / parser / decoder / compiler 的显式分支

### 5.4 输入抽象容易

优先找不需要解析复杂文件格式内部细节，也能表达触发性的案例：

- 可以抽象成“攻击者可提供特制 regex pattern”
- 可以抽象成“攻击者可提供 gzip / webp / tar / git revspec / SQL / XML 输入”
- 可以抽象成“大长度 / 大容量 / 特定标志位已设置”的输入假设

也就是说：

- **输入内容可以复杂**
- 但工具最好只需要知道“输入类别和关键前置约束”，不需要完整理解整个文件格式

### 5.5 可解释性强

优先找能回答以下问题的案例：

- 具体是哪条调用链走到了漏洞相关 sink
- 哪些参数或配置满足了触发条件
- 哪个版本是脆弱版本
- 如果没触发，是因为缺了哪条条件

---

## 6. 明确不要的候选

请主动过滤以下项目或漏洞：

- 只能证明“依赖存在”，无法证明路径或条件
- 真实用的是系统库，但系统库版本拿不到
- 需要极重的动态环境模拟才有意义
- 漏洞触发点在 native 内部极深处，Rust 侧只有一个完全无语义的 opaque 调用
- 项目几乎无法构建，连 `cargo metadata` 或目标入口都定位不了
- 需要深度解析专有协议/复杂二进制格式，且 Rust 侧没有任何前置条件线索
- 触发前提完全依赖运行时网络状态、操作系统配置、远端服务行为，静态上无可靠依据
- 漏洞影响版本、修复版本、组件映射关系都不清楚

---

## 7. 希望 agent 如何搜索

请按下面的顺序搜索，不要只做关键词搜索后随便给结论。

### 第一步：先找“漏洞组件”

优先搜索这些类别：

- `zlib`
- `libwebp`
- `libgit2`
- `pcre2`
- `sqlite`
- `openssl`
- `libxml2`
- `expat`
- `libarchive`
- `git2` 生态常见 bundled native
- 图像解码、压缩解压、正则、解析器、数据库、版本控制相关 native 组件

优先找：

- 有明确 CVE
- 有明确触发条件
- 有明确版本范围

### 第二步：再找“真实 Rust 项目”

搜索这些项目特征：

- 依赖了对应 Rust crate 或 `*-sys` crate
- 有真实 CLI / library 使用场景
- 有 release/tag/lockfile
- 项目不是太小，也不是只有演示代码

### 第三步：核对“是否适合我的工具”

至少确认以下问题：

- 能否明确解析依赖链
- 能否生成 Rust CPG
- Rust 侧能否识别到至少一个高价值 sink
- 触发条件是否能在 Rust 侧看到一部分
- 是否能构造成规则，而不是只能人工阅读解释

## 8. 每个候选必须交付的信息

对每个推荐候选，必须按下面格式输出，不要只给链接。

### 8.1 基本信息

- 项目名称
- 项目仓库地址
- 具体版本 / tag / commit
- 组件名称
- CVE 编号
- 漏洞后果（崩溃、越界写、RCE、DoS、信息泄露等）

### 8.2 依赖与版本证据

- Rust 依赖链：`root -> ... -> crate -> sys crate -> native`
- 是否 bundled / vendored / system
- 实际锁定版本
- 版本证据来自哪里：
  - `Cargo.lock`
  - `Cargo.toml`
  - build.rs
  - vendor 目录
  - 文档 / 发布说明

### 8.3 漏洞触发条件

必须拆成结构化条件，例如：

- 条件 1：脆弱版本存在
- 条件 2：必须走到 WebP / gzip / PCRE2 / libgit2 这条分支
- 条件 3：输入对象必须能流到某个 sink 参数
- 条件 4：某些 builder flag / mode / feature 必须开启
- 条件 5：必须存在某个 API 调用序列
- 条件 6：长度 / 容量 / header buffer 等约束必须满足

### 8.4 Rust 侧证据

请指出：

- 哪些 Rust 函数是关键入口
- 哪些 Rust API / builder / wrapper 是关键 sink
- 哪些参数、字段、分支、flag 在 Rust 侧可见
- 哪些条件在 Rust 侧看不到，只能作为输入假设

### 8.5 适配当前工具的可行性分析

请明确判断：

- 是否适合当前工具
- 为什么适合
- 难点在哪里
- 需要补什么能力
- 是否更适合作为：
  - confirmed 候选
  - TriggerableWithInputAssumption 候选
  - 仅 reachable 候选

### 8.6 最终结论

每个候选都要给一个结论标签：

- **强烈推荐**
- **可做但需要补能力**
- **不推荐**

并说明理由。

请按下面格式输出最终结果。

### 9.1 先给总表

先用表格列出每个候选的：

- 项目
- CVE
- native 组件
- 漏洞后果
- 版本是否可确认

### 9.2 再给候选详情

每个候选至少包含：

1. 项目简介
2. 漏洞简介
3. 依赖链与版本证据
4. Rust 侧关键路径
5. 触发条件拆解
6. 适配当前工具的优点
7. 可能卡住的点

### 9.3 最后给排序建议

请把候选按以下优先级排序：

1. 最适合马上接入工具

3. 研究价值有，但当前阶段不建议优先做

## 10. 推荐的评估标准

请用 100 分制给每个候选打分，并拆分评分项：

- 版本证据清晰度：20 分
- Rust 侧 sink / 条件可见性：20 分
- CPG 生成可行性：20 分
- 触发条件规则化难度：15 分
- 输入可抽象程度：10 分
- 可解释性：10 分
- 辅助资料完整度：5 分

评分时请解释扣分原因，不要只给总分。

## 11. 可以参考的正例模式

下面这些模式通常是好候选：

### 11.1 解码器 / 解析器类

- Rust 代码先根据扩展名、格式标记、magic bytes、配置分支进入某个 decoder
- 输入字节从 `File::open` / `read_to_end` / buffer / request body 流到 decoder
- 后续调用 `.decode()` / `.parse()` / `.read_*()` / `.inflate()` / `.compile()`
- 有明确的版本约束和 feature 开关

### 11.2 builder / flag 组合类

- Rust 代码上能看到 `caseless`、`utf`、`ucp`、`jit`、`strict`、`unsafe` 等 flag
- 漏洞需要多个 flag 同时满足
- 非常适合当前工具做条件建模

### 11.3 API 序列类

- 漏洞需要先初始化对象、再设置字段、再执行某个方法
- 例如：
  - 先 `inflateGetHeader()`
  - 再多次 `inflate()`
  - 且 header buffer 与实际输入长度有关系

### 11.4 Git / archive / database / regex 类

- 这几类通常更容易在 Rust 侧看到入口、参数和配置
- 也更容易形成“输入 -> 中间对象 -> sink”的规则

---

## 12. 需要 agent 特别注意的误区

请不要犯下面这些错误：

- 只因为项目依赖了漏洞组件，就断言“可触发”
- 只因为项目编译失败，就断言“不适合”
- 不区分 direct dependency、transitive dependency、sys crate、native 组件
- 不区分 bundled 和 system library
- 不区分 reachable、triggerable、TriggerableWithInputAssumption
- 不解释触发条件，只给“这个 CVE 很危险”
- 不说明 Rust 侧是否真的看得到关键条件

---

## 13. 我更关心什么

如果你要在“候选数量”和“候选质量”之间做取舍，请优先保证质量。

我更关心：

- 这个项目是否真的适合当前工具
- 为什么适合
- 为什么不适合
- 规则该怎么写
- 哪些条件已经能静态看到
- 哪些条件只能作为输入假设
- 最终能否给出清晰、可信、可解释的结论

我不需要：

- 大量泛泛而谈的 CVE 列表
- 没有版本证据的项目清单
- 只有“可能有风险”的模糊判断

## 14. 可直接复制给其他 Agent 的任务提示词

下面这段可以直接复制给其他 agent：

> 请帮我寻找一批适合“Rust 项目依赖 native 组件的 n-day 供应链漏洞可达/可触发检测”的真实案例。  
>  
> 要求你筛选的不是泛泛有漏洞的项目，而是适合静态规则 + CPG 检测落地的项目。请优先寻找 Rust 项目通过 `*-sys` crate、FFI crate、bundled/vendored 方式依赖 C/C++ 组件的案例。漏洞必须是公开已知 CVE，且版本范围尽量明确。  
>  
> 你需要重点判断：  
> 1. 项目是否是真实 Rust 项目，源码、版本、tag、lockfile 是否可获得。  
> 2. 是否能明确依赖链：`Rust 项目 -> Rust crate -> sys crate -> native 组件`。  
> 3. 是否能明确 bundled / vendored / system library，以及实际锁定的 native 版本。  
> 4. Rust 侧是否能看到至少一个高价值 sink / wrapper / builder / decode / parse / compile 路径。  
> 5. 漏洞触发条件是否能拆成“版本 + 路径 + 参数/flag/分支/状态/长度/API 序列/input assumption”的结构化规则。  
> 6. 是否适合当前静态工具：当前工具依赖 Rust CPG，因此没有可生成的 Rust CPG 就不算合格候选。  
> 7. 是否存在明显的输入对象流向，例如 path、bytes、regex pattern、header buffer、SQL、Git revspec、压缩流、图片字节等。  
> 8. 哪些条件在 Rust 侧可见，哪些条件只能作为攻击输入假设。  
>  
> 请优先推荐版本证据清晰、Rust 侧条件可见、CPG 可生成、规则容易解释的候选。  
>  
> 不要推荐以下情况：  
> - 只能证明依赖存在，无法证明路径和条件  
> - 真实使用的是系统库，但版本无法确认  
> - 漏洞触发点完全藏在 native 内部，Rust 侧没有任何前置语义  
> - 无法恢复 Cargo 编译上下文，几乎不能生成 Rust CPG  
> - 需要重度动态环境、网络环境或复杂协议内部解析才有分析价值  
>  
> 请输出：  
> - 一个候选总表（项目 / CVE / native 组件 / 漏洞后果 / 版本是否可确认 / Rust sink 是否清晰 / CPG 是否大概率可生成 / 推荐等级）  
> - 每个候选的详细分析：  
>   - 项目名称、仓库、版本  
>   - CVE、native 组件、漏洞后果  
>   - 依赖链与版本证据  
>   - Rust 侧关键函数 / sink / 配置路径  
>   - 漏洞触发条件拆解  
>   - 哪些条件 Rust 侧可见，哪些只能作为输入假设  
>   - 适配当前工具的优点、难点、是否需要补能力  
>   - 最终结论：强烈推荐 / 可做但需要补能力 / 不推荐  
> - 最后按优先级排序，并给每个候选 100 分制评分，说明扣分原因。  
>  
> 我更看重候选质量、规则可解释性、版本证据和工具落地可行性，不要只给一堆泛泛的漏洞列表。

## 15. 补充说明

如果 agent 发现某个漏洞很有价值，但当前工具还缺一个关键能力，也可以保留这个候选，但必须写清楚：

- 当前缺的能力是什么
- 为什么这项能力值得补
- 补完之后这个案例为什么会变成强候选

这样我可以区分：

- 当前就能做的
- 需要小改动就能做的
- 研究价值有，但当前不应投入的
